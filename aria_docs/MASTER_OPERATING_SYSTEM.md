6. Transcription extracted and cleaned
7. Claude Haiku generates actionable summary
8. MP3 audio fetched from ElevenLabs
9. Resend delivers formatted email to business owner (summary + transcript + audio)
10. Everything logged to Supabase

- **P**ain: What specific problem are they trying to solve?

**Disqualify if:**
- Fewer than 5 calls per day (too low volume to justify cost)
- Already has a working solution they're happy with
- Budget is under $100/mo with no flexibility
- No authority to make decisions

## 6.4 The 15-Minute Demo Framework

```
MINUTE 0-2: OPENING
"Thanks for taking the time, [Name]. Before I show you anything,
tell me — how do you handle calls right now when you're busy?"
→ LISTEN. Take notes. They'll reveal their pain.

MINUTE 2-7: LIVE DEMO
"Let me show you something. I'm going to call this number on speaker."
→ Call demo line. Walk through realistic scenario.
→ Show email summary arriving 30 seconds later.
→ This is the "wow" moment.

MINUTE 7-12: ROI CONVERSATION
"Based on what you told me — [X] calls/day, missing [Y] of them,
at $[Z] average ticket — that's $[math]/month in missed revenue.
This service costs $[price]/month. It pays for itself if it
saves you [small number] customers."

MINUTE 12-15: CLOSE
"Here's what I'd like to do: set this up for you at no cost for
two weeks. You try it, your customers try it. If it's not making
your life easier, I turn it off. Sound fair?"
→ STOP TALKING. Let them respond.
```

**Tone:** Confident, warm, direct. Like a smart friend who happens to know a lot about AI.

**Don't:**
- Use jargon with non-technical audiences
- Overpromise ("guaranteed results")
- Bash competitors by name in public content
- Use AI hype language ("revolutionary," "game-changing," "disrupting")

## 11.2 Incident Response Playbook

```

5. RESOLVE — Fix the root cause
6. COMMUNICATE — Tell the client what happened, what we did, how we prevented recurrence
7. LOG — Document everything in CertusOrdo audit trail
8. LEARN — Update knowledge base to prevent recurrence
```

# 13. CERTUSRODO GOVERNANCE RULES
# ═══════════════════════════════════════════════════════════════════

## 13.1 Transaction Logging

Every action Aria takes is wrapped in a CertusOrdo transaction:

```
1. BEGIN transaction with pre-state snapshot
   - What is the current state before this action?
   - What action is being taken?
   - What is the expected outcome?

2. EXECUTE the action
   - Send the email, post the content, update the record

3. CAPTURE post-state
   - What changed?
   - Was the outcome expected?

4. COMMIT or ROLLBACK
   - If outcome matches expectation: COMMIT
   - If any rollback trigger fired: ROLLBACK
   - If uncertain: HOLD FOR REVIEW (flag to Ian)
```

## 13.2 Audit Trail Requirements

Every logged transaction must include:
- Timestamp (UTC)
- Action type (email_send, content_post, client_checkin, lead_qualification, etc.)
- Target (who/what was acted upon)
- Pre-state (what existed before)
- Post-state (what exists after)
- Outcome (success, failure, pending)
- Confidence score (0-100)
- Rollback eligibility (true/false)

## 13.3 Confidence Thresholds

| Confidence | Action |
|-----------|--------|
| 90-100% | Execute autonomously, log normally |
| 70-89% | Execute, flag to Ian in daily digest |
| 50-69% | Execute with caution, flag immediately |
| Below 50% | Do NOT execute. Escalate to Ian. |
